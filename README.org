#+title: org-defuddle

~org-defuddle~ is intended to be a Rust rewrite of [[https://github.com/kepano/defuddle][kepano/defuddle]] with an
Emacs dynamic module frontend that inserts extracted readable content into Org.

The current reference snapshot inspected during scaffolding is upstream commit
~19520ad~. The long-term goal is feature parity with defuddle's extraction
pipeline: metadata, schema.org fallback, DOM cleanup, scoring, site-specific
extractors, media handling, footnotes, math/code normalization, and Org content
conversion.

* Architecture

- ~crates/org-defuddle-core~: Rust extraction core. It owns HTML parsing, DOM
  cleanup, metadata extraction, content scoring, and HTML-to-Org conversion.
- ~crates/org-defuddle-module~: Emacs dynamic module exposing coarse-grained
  calls into the Rust core.
- ~crates/org-defuddle-cli~: Rust command-line wrapper for URL, file, and stdin
  parsing with Org default output plus JSON, frontmatter, property-style, and
  compatibility Markdown output.
- ~org-defuddle.el~: Emacs package frontend for loading the module,
  fetching URLs, parsing buffers, and inserting Org.

The dynamic module boundary is deliberately coarse:

#+begin_src emacs-lisp
(org-defuddle-parse-html html url)
;; => (:title ... :url ... :word_count ... :html ... :org ...)

(org-defuddle-parse-html html url '(:include-images nil))
;; => same plist shape, with content images omitted from :html and :org

(org-defuddle-parse-html html url '(:remove-small-images nil))
;; => same plist shape, preserving small icons and unresolved placeholders

(org-defuddle-parse-html html url '(:content-selector "article.post"))
;; => same plist shape, using the matching element as main content

(org-defuddle-parse-html html url '(:include-replies nil))
;; => same plist shape, omitting supported replies/comments

(org-defuddle-parse-html html url '(:remove-hidden-elements nil))
;; => same plist shape, preserving display:none/hidden content

(org-defuddle-parse-html html url '(:remove-exact-selectors nil))
;; => same plist shape, preserving exact selector matches such as footers

(org-defuddle-parse-html html url '(:remove-partial-selectors nil))
;; => same plist shape, preserving partial selector matches such as related blocks

(org-defuddle-parse-html html url '(:remove-content-patterns nil))
;; => same plist shape, preserving read-time/related-card pattern content

(org-defuddle-parse-html html url '(:remove-low-scoring nil))
;; => same plist shape, preserving blocks removed by low-score heuristics
#+end_src

* Installation

Requirements:

- Emacs 27.1 or newer with dynamic module support.
- macOS, GNU/Linux, or Windows on a supported architecture.

Install the Elisp package from the repository, then run an interactive
org-defuddle command:

#+begin_src emacs-lisp
(use-package org-defuddle
  :vc (:url "https://github.com/LuciusChen/org-defuddle"
       :rev :newest))
#+end_src

On first interactive use, org-defuddle offers to download the pinned pre-built
Rust module from GitHub Releases.  The module is installed under
~user-emacs-directory/modules/~, verified against the release ~SHA256SUMS~
manifest, and loaded immediately.  Installation can also be triggered
explicitly:

#+begin_src emacs-lisp
M-x org-defuddle-download-module
#+end_src

Release binaries are published for:

- ~aarch64-apple-darwin~ (Apple Silicon)
- ~x86_64-apple-darwin~ (Intel macOS)
- ~aarch64-unknown-linux-gnu~
- ~x86_64-unknown-linux-gnu~
- ~x86_64-pc-windows-msvc~

The downloaded filename is normalized to ~liborg_defuddle_module~ plus the
platform's ~module-file-suffix~.  Customize ~org-defuddle-module-file~ to use a
different persistent location.

* Building From Source

Building requires the Rust toolchain and is only needed for development or an
unsupported release platform.

#+begin_src sh
cargo build --release -p org-defuddle-module
cargo build --release -p org-defuddle-cli
#+end_src

The compiled module is expected at one of:

- macOS: ~target/release/liborg_defuddle_module.dylib~
- GNU/Linux: ~target/release/liborg_defuddle_module.so~
- Windows: ~target/release/org_defuddle_module.dll~

The CLI binary is built at ~target/release/org-defuddle~.

Repository-local release builds are detected as a development fallback when no
downloaded module exists.  Tagged GitHub Releases also contain pre-built CLI
assets named ~org-defuddle-cli-<target>~.

* Emacs Usage

#+begin_src emacs-lisp
(add-to-list 'load-path "~/repos/org-defuddle")
(require 'org-defuddle)

(org-defuddle-url-to-org "https://example.com/post")

* CLI Usage

The Rust CLI supports upstream-style ~parse~ options for URLs, local HTML
files, and stdin:

#+begin_src sh
target/release/org-defuddle parse https://example.com/post
target/release/org-defuddle parse article.html
target/release/org-defuddle parse article.html --json
target/release/org-defuddle parse article.html --markdown
target/release/org-defuddle parse article.html --separate-markdown --json
target/release/org-defuddle parse article.html --frontmatter
target/release/org-defuddle parse article.html --property title
target/release/org-defuddle parse article.html --property contentMarkdown
target/release/org-defuddle parse article.html --output article.org
target/release/org-defuddle fetch https://example.com/api --header 'Accept: application/json'
#+end_src

CLI URL fetching includes a defuddle-style user agent, optional
~--user-agent~, optional ~--lang~ as ~Accept-Language~, HTML content-type
validation, a 5MB response limit, redirects, charset decoding, proxy
environment variables, and ~NO_PROXY~ matching. The ~fetch~ subcommand exposes
the same fetch stack for JSON/XML/text API helper requests with custom method,
headers, body, user agent, and language. Full Turndown parity remains tracked as
a parity gap. If a URL extraction is empty and the user did not set
~--user-agent~, the CLI retries once with defuddle's bot user agent and can
recover raw Markdown from the pre-DOM response body.

Images are included by default.  To omit images from extracted Org output:

#+begin_src emacs-lisp
(setq org-defuddle-include-images nil)

;; Or override per call:
(org-defuddle-html-to-org html "https://example.com/article" '(:include-images nil))

;; Upstream-compatible inverse spelling:
(org-defuddle-html-to-org html "https://example.com/article" '(:remove-images t))
#+end_src

Small images and unresolved image placeholders are removed by default.  To keep
them:

#+begin_src emacs-lisp
(setq org-defuddle-remove-small-images nil)

;; Or override per call:
(org-defuddle-html-to-org html
                          "https://example.com/article"
                          '(:remove-small-images nil))
#+end_src

To bypass automatic main-content detection and use a specific CSS selector:

#+begin_src emacs-lisp
(setq org-defuddle-content-selector "article.post-content")

;; Or override per call:
(org-defuddle-html-to-org html
                          "https://example.com/article"
                          '(:content-selector "main article"))
#+end_src

Site-specific replies and comments are included by default.  To omit them:

#+begin_src emacs-lisp
(setq org-defuddle-include-replies nil)

;; Or override per call:
(org-defuddle-html-to-org html "https://example.com/post" '(:include-replies nil))
#+end_src

API transcript extractors use a stable default language order.  To prefer a
specific language where the site exposes multiple tracks:

#+begin_src emacs-lisp
(setq org-defuddle-language "en")

;; Or override per URL call:
(org-defuddle-url-to-org "https://www.bilibili.com/video/BV1nNkFBMEDS"
                         '(:language "zh-cn"))

(org-defuddle-url-to-org "https://www.youtube.com/watch?v=dQw4w9WgXcQ"
                         '(:language "en"))
#+end_src

The language option ranks subtitle tracks already exposed by the video service;
it does not transcribe audio.  When a YouTube video has no caption track,
extraction falls back to the video metadata and description without a
transcript.

URL extraction uses supported site APIs and alternate hosts by default for
content that is not usually present in raw HTML, such as YouTube/Bilibili
transcripts and Reddit comments.  To force the original URL HTML path:

#+begin_src emacs-lisp
(setq org-defuddle-use-async nil)

;; Or override per URL call:
(org-defuddle-url-to-org "https://www.youtube.com/watch?v=dQw4w9WgXcQ"
                         '(:use-async nil))
#+end_src

HTML standardization is enabled by default.  To preserve more of the original
HTML structure before Org rendering:

#+begin_src emacs-lisp
(setq org-defuddle-standardize nil)

;; Or override per call:
(org-defuddle-html-to-org html
                          "https://example.com/article"
                          '(:standardize nil))
#+end_src

Debug and profiling diagnostics are disabled by default.  To include upstream
style debug/profile fields in JSON output:

#+begin_src emacs-lisp
(org-defuddle-parse-html html
                         "https://example.com/article"
                         '(:debug t :profile t))
#+end_src

YAML frontmatter is omitted by default.  To prepend upstream-style frontmatter
to Org output:

#+begin_src emacs-lisp
(setq org-defuddle-frontmatter t)

;; Or override per call:
(org-defuddle-html-to-org html
                          "https://example.com/article"
                          '(:frontmatter t))
#+end_src

To extract a single upstream-style response property:

#+begin_src emacs-lisp
(org-defuddle-parse-html-property html
                                  "title"
                                  "https://example.com/article")

(org-defuddle-parse-html-property html
                                  "wordCount"

;; Or override per call:

                          "https://example.com/article"

                            :remove-partial-selectors nil))
#+end_src

Content-pattern cleanup is enabled by default.  To preserve read-time labels,
related sections, and similar pattern-matched blocks:

#+begin_src emacs-lisp

;; Or override per call:
(org-defuddle-html-to-org html
                          "https://example.com/article"
                          '(:remove-content-patterns nil))
#+end_src

Low-scoring block removal is enabled by default.  To preserve blocks that look
like navigation, recommendation, or listing clutter:

#+begin_src emacs-lisp
(setq org-defuddle-remove-low-scoring nil)

;; Or override per call:
(org-defuddle-html-to-org html
                          "https://example.com/article"
                          '(:remove-low-scoring nil))
#+end_src

For local HTML:

#+begin_src emacs-lisp
(with-temp-buffer
  (insert-file-contents "article.html")
  (org-defuddle-buffer-to-org "https://example.com/article"))
#+end_src

* Current Status

This repository targets Defuddle extraction parity with Org as the canonical
output. It does not target byte-for-byte Markdown output or the JavaScript
public API. The fixture harness covers all 202 upstream fixture HTML files in
the inspected defuddle snapshot. Every fixture strictly matches the upstream
expected ~title~, ~author~, ~site~, and ~published~ metadata, alongside
fixture-specific content and cleanup assertions. All 202 final upstream content
fragments also match the Rust output exactly after both are rendered through
the same Org renderer. Every generated Org document is compared byte-for-byte
with its checked-in snapshot under
~tests/expected-org/~. Those Org snapshots are generated by org-defuddle and
are regression baselines, not independent upstream-derived parity evidence.

Regenerate the checked-in Org snapshots explicitly after an intentional output
change:

#+begin_src sh
ORG_DEFUDDLE_UPDATE_EXPECTED_ORG=1 cargo test -p org-defuddle-core \
  --test upstream_fixtures all_upstream_fixture_org_regression_snapshots
#+end_src

For an independent exact Org differential, first build the pinned upstream Node
entry point, then compare every upstream final content fragment and every Rust
final content fragment through the same Org renderer:

#+begin_src sh
cd /path/to/defuddle && npm ci && npm run build:node
cd /path/to/org-defuddle
ORG_DEFUDDLE_DEFUDDLE_DIR=/path/to/defuddle cargo test -p org-defuddle-core \
  --test upstream_fixtures all_upstream_fixture_content_matches_exact_org_rendering \
  -- --ignored
#+end_src

This differential test is intentionally ignored in the default suite because
it requires a separately installed and built upstream Node checkout. The 202
checked-in Org snapshots remain the normal offline regression gate.

Set ~ORG_DEFUDDLE_DIFF_FIXTURE=<fixture-name>~ on the same command to print the
exact expected and actual Org for one mismatch.

Known remaining gaps:

- Org is the canonical output. First-pass Markdown exists only for upstream
  compatibility through ~contentMarkdown~, ~separateMarkdown~,
  ~:separate-markdown~, and CLI ~--separate-markdown~/~--markdown~; full
  Turndown parity is intentionally outside the Org-parity completion gate. The
  Rust CLI covers URL/local HTML/stdin parsing with JSON, YAML frontmatter,
  output-file, debug, property-style options, and compatibility Markdown
  output.
- Debug/profile fields are exposed for the Rust core, dynamic module, and Elisp
  JSON path, and report removals from hidden-element, upstream-style
  ~removeBySelector~ selector cleanup,
  upstream-style ~scoreAndRemove~ low-scoring cleanup, and upstream-style
  ~removeByContentPattern~ cleanup. Coverage is still narrower than upstream's
  full debug removals. Profile JSON uses the upstream generic-pipeline stage
  names plus the standardization and ~se:~ rule substages selected by the
  ~standardize~ and ~debug~ options. Serialized shadow bridges and inline mobile
  media rules are processed;
  browser-only state that was never serialized cannot be recovered.
  Site-specific extractor responses omit debug/profile data as upstream does.
  Debug JSON uses upstream ~contentSelector~ naming and retains
  ~content_selector~ as an org-defuddle compatibility alias.
- Browser-only extractor actions are not fully replicated. In particular,
  upstream YouTube extraction can click the desktop or mobile transcript panel
  and wait for dynamically rendered segments. org-defuddle supports existing
  desktop/mobile transcript DOM, inline ~ytInitialPlayerResponse~ caption
  tracks, and Innertube timedtext, but it cannot recover a transcript panel that
  is absent from serialized HTML and requires JavaScript interaction.

Implemented:
